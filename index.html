Chapter 1
System Overview
What the device does, and where it fits in the Telstra DCIM fabric.

1.1 What is SM5000?

The SM5000 is a small PoE-powered controller that lives in a data-hall rack and reports back-of-rack temperature and humidity into Telstra's Fleet Manager (FM) platform. Each unit is a single ESP32 + two RS-485 probes deployment that publishes telemetry over MQTT, serves a Modbus TCP slave endpoint (for DCIM/SCADA integration), and exposes a local HTTP UI for on-site configuration.

The firmware is self-contained and field-upgradable over the air. Once deployed and joined to Fleet Manager, a unit is expected to run unattended for its service life with only occasional calibrations and no routine site visits.

1.4Feature list
  • Two-sensor RS-485 Modbus RTU acquisition with per-sensor calibration offsets.
  • Third-sensor reference-probe input for in-field calibration (hot-plug detection).
  • Dual-core firmware: dedicated MQTT core, dedicated data-pipeline core, separate UI loop.
  • Modbus TCP slave exposing 9 holding registers including a status bit-field.
  • HMAC-SHA256 signed control plane (separate keys for OTA, config, and FM update).
  • Seven-level MQTT topic hierarchy (country → datahall → device) so commands can be scoped to any rack, row, hall, DC, city, or country.
  • Staggered, group-based fleet OTA and fleet reboot (1…6 groups, MAC-derived).
  • NVS-persisted OTA result broadcast — the device confirms the outcome of its last OTA on the next successful MQTT connect after reboot.
  • Hybrid time source: FM (preferred) → NTP → internal clock, with a 4-state sync machine that gates logging.
  • Circular SPIFFS log holding ≈ 7 days at 1-minute resolution, with a 96-point downsampled chart for the web UI.
  • Front-panel button UX for calibration and factory reset.
  • Multiple independent watchdogs: Task WDT, heap threshold, link/gateway ping, scheduled 6 h preventive reboot, PubSubClient dead-socket workaround.
1.5Non-goals
Out of scope: the Fleet Manager backend, DCIM client software, and the sensor probes' own firmware — these are upstream systems from the SM5000's perspective.
  • Not supported: Wi-Fi operation. The device is Ethernet-only (PoE). The Wi-Fi stack headers are present only because PubSubClient/WiFiClient glue types are shared between Ethernet and Wi-Fi in the Arduino-ESP32 core.
  • Not supported: TLS-encrypted MQTT. The firmware uses plain MQTT (port 1883 by default) behind the carrier-managed network; command authentication is provided by HMAC signing on the application payload, not by transport-layer TLS. (HTTPS is used for OTA firmware download.)
  • Not targeted: ESP32-S2, -S3, -C3 variants. Only classic ESP32 is supported because the build pins to Arduino-ESP32 core 2.0.14.

Hardware Reference
Every electrical interface the firmware expects, and the non-obvious electrical behaviours it depends on.
2.1Board summary

The SM5000 carrier mounts an ESP32-WROOM module, a Microchip LAN8720 10/100 Ethernet PHY (RMII), a PoE front end supplying the 5 V rail, two isolated MAX485 RS-485 transceivers (one per UART bus), a PCF8574-backed 16×2 I²C LCD, and a tactile front-panel button. All firmware pin assignments are centralised in Config.h §2.

ESP32-WROOM & the external PHY. The ESP32-WROOM module contains an internal Ethernet MAC (EMAC) controller, but it does not contain a Physical-layer transceiver. The external LAN8720 chip provides only the PHY layer (the line driver, auto-negotiation, and cable-side signalling); the MAC sits inside the ESP32 and communicates with the PHY over the RMII bus (GPIO17/18/23 + the fixed RMII data pins). This is why "the ESP32 supports Ethernet" still requires an external PHY chip on the board.
2.2Power tree
The SM5000 does not supply power to the RS-485 sensors. Sensors 1, 2, and the optional reference probe are powered from a separate 24 V DC site rail (shown in the lower yellow box above). The SM5000 carries only RS-485 data wires (A/B + common) to each sensor — it does not carry sensor power. Installers must provision the 24 V rail separately.
  • GPIO12 = PoE 5 V enable and PHY power gate. Both POE_5V_ENABLE_PIN and ETH_PHY_POWER resolve to GPIO12.
  • Firmware drives GPIO12 HIGH in setup() and leaves it HIGH for the device's entire run.
  • The PHY's nRST is handled purely in hardware by an RC network (~100 ms time constant). Firmware does not drive nRST (ETH_PHY_RST = -1). The 500 ms delay() after raising GPIO12 lets this RC settle before MDIO transactions begin.
⚠ Do not re-purpose GPIO12 at runtime. Toggling it low will power-cycle the PHY and the PoE output stage simultaneously.

2.4RS-485 topology

The firmware runs two independent Modbus RTU masters, one per UART, each with its own DE/RE transceiver.

2.4.1 UART2 — main sensors
  • Baud: 9600 8N1 (MODBUS_BAUD). Slaves: Sensor 1 (SENSOR1_ID = 1), Sensor 2 (SENSOR2_ID = 2).
  • Registers read: humidity at 0x0000, temperature at 0x0001, both holding, both scaled ÷ 10.
  • Inter-frame gap: 500 ms delay() between sensor 1 and sensor 2.
2.4.2 UART1 — reference / calibration probe
  • Same baud, same register layout. Slave ID CALIBRATION_SENSOR_ID = 3.
  • Hot-plug detected — isCalibrationSensorConnected() decides whether the button UX offers calibration or factory reset.
2.4.3 DE/RE timing

preTransmission() drives DE/RE HIGH before every outbound frame. postTransmission() inserts a delayMicroseconds(200) before dropping DE/RE LOW. This 200 µs tail protects the stop bit — do not reduce it without re-measuring on a scope.

2.8Known hardware caveats
  1. GPIO12 double duty — treat as power-rail control, not general GPIO.
  2. GPIO34 is input-only with no internal pull-down; INPUT_PULLUP, button shorts to GND.
  3. PHY rail has no soft-start regulator — 500 ms boot delay is the only guard against MDIO-before-ready.
  4. No external watchdog IC — all self-healing is firmware-level.
  5. Both RS-485 buses share the same bit rate.

Chapter 4
Dual-Core Runtime Model
Which work runs on which core, why, and how the two cores coordinate.
4.1Partitioning philosophy

The ESP32 exposes two cores. We allocate them by blocking character, not by feature:

  • Core 0 runs anything that can stall on a socket — the MQTT listener.
  • Core 1 runs the deterministic data pipeline — sensor acquisition, Modbus TCP service, DataLogger writes, MQTT publish.
  • The Arduino loop() (FreeRTOS schedules it on Core 1 at priority 1) owns UI and non-critical housekeeping.
The inversion of the conventional "loop runs everything" Arduino pattern is deliberate and load-bearing. Do not move MQTT service back into loop() without re-deriving the 60 s keep-alive timing margin.

The priority ordering 6 > 4 > 1 is by design — MQTT cannot be starved by the pipeline, and the pipeline cannot be starved by the UI loop.

The chosen values are intentionally non-consecutive (6, 4, 1 rather than 3, 2, 1). The gaps are deliberate: they leave headroom for future tasks to be inserted at an intermediate priority without having to re-tune every existing task. For example, a future "security event" task could sit at priority 5 (just below MQTT but above the pipeline) without forcing a priority-number cascade across all other tasks.

4.3Core 0 — MQTT Listener
4.3.1 Lifecycle & preconditions

Key numeric gates: 10 s stabilisation delay before any MQTT activity; 3 s TCP connect timeout via ethClient.setTimeout(3); 60 s keep-alive (client->setKeepAlive(60)); and a 2048-byte PubSubClient buffer set via client->setBufferSize(2048) to accommodate the largest inbound payloads (OTA commands carrying URL + SHA-256 checksum + metadata). This is separate from the publish-side mqttPublishBuffer (MQTT_PAYLOAD_SIZE = 512 B) used for outbound telemetry.

4.3.2 Reconnect state machine
  • Reconnect attempts throttled to one per MQTT_RECONNECT_INTERVAL = 60 s.
  • Before every reconnect, ethClient.stop() closes any stale socket — see §10.7.
  • If link is down (!ETH.linkUp() || localIP() == 0.0.0.0), skip the attempt and delay 2 s.
  • All PubSubClient state codes are logged on reconnect (-4 CONNECTION_TIMEOUT through 5 UNAUTHORIZED).
4.3.3 Keep-alive handling

Every tick (10 ms) while connected, mqttClientMutex is taken with 100 ms timeout and client->loop() is called. A hard tick is additionally forced every MQTT_KEEPALIVE_INTERVAL = 15 s. Sustained mutex misses will cause the broker to time out keep-alive (60 s budget).

4.4Core 1 — Data Pipeline
4.4.1 Warm-up & readiness gates
  • 5 s initial sleep before the first iteration.
  • One immediate readCalibrationSensor() at startup.
  • Per-iteration check: if otaInProgress is set, the task sleeps 1 s and continues — it is not suspended at this point. This is a soft gate — the task stays alive, feeds the Task WDT, but skips its work cycle.
  • A second, harder pause exists for actual flash writes: stopDataPipeline() calls vTaskSuspend(dataPipelineTaskHandle) (see §10.5 and §7.5.1). Both mechanisms exist together by design — the soft gate gives the task an orderly skip-and-yield path before the hard suspend freezes it.
4.4.2 Sensor poll cycle (every 10 s)
  1. readSensorsAndUpdateLKG() — reads both Modbus RTU sensors, applies offsets, updates LKG only on non-zero.
  2. readCalibrationSensor() — reads reference probe on UART1.
  3. If calibration is running, processCalibrationSample() is invoked internally.
4.4.3 Modbus TCP gate

Until both LKG temperature slots are non-zero, the Modbus TCP server is not started. Once the condition is met, initModbusTCP() runs once, modbusStarted latches true, and handleModbusTCP() runs every iteration afterwards. DCIM clients get ECONNREFUSED rather than a misleading 0.0 °C.

4.4.4 MQTT publish & DataLogger paths
  • Every MQTT_PUBLISH_INTERVAL = 30 s, publishSensorData(t1, h1, t2, h2) runs if enabled + connected.
  • DataLogger writes are gated on: sensors read at least once, and dataLogger.shouldLog() returning true (interval + time-valid). 5 s back-off on failed writes.
  • timeManager.checkStateTransitions() runs every iteration.
4.5Arduino loop()
  1. Feed Task WDT if ≥ 5 s since last feed.
  2. processAsyncConfigUpdate() — advance 8-state FSM (Appendix K).
  3. checkSystemHealth() — heap, link, gateway, uptime, scheduled-reboot logic.
  4. handleOTA() + processPendingOTA() — fire staged OTAs when stagger elapses.
  5. If otaInProgress, delay(100) and return.
  6. handleCalibrationSystem() → processButtonEvents().
  7. checkInactivityTimeout() — auto-exit after 3 s.
  8. LCD refresh 1 Hz (unless calibrating or OTA).
  9. If shouldReboot, 3 s countdown then ESP.restart().
  10. delay(10).
4.6Event-driven workers
4.6.1 AsyncOTA task

Created on demand by startOTAFromMQTT(), startFleetManagerUpdateAsync(), or startManualUpdate(). One-shot; terminates on success (reboot) or failure (resumes the pipeline and posts status).

4.6.2 AsyncTCP workers (Modbus TCP & web server)

Event-driven callbacks owned by AsyncTCP / ESPAsyncWebServer. They don't compete with named tasks for CPU. Long downloads (CSV export) are chunked at ~30 KB (DATA_LOG_CSV_CHUNK_ENTRIES = 1250).

4.6.3 GPIO34 Button Handler (ISR + deferred)
IRAM (Internal RAM) is a fast, instruction-accessible memory region on the ESP32 into which code can be placed with the IRAM_ATTR attribute. Code in IRAM executes even while flash is busy (for example, during an SPI-flash read or SPIFFS write). Interrupt service routines must be IRAM-resident, otherwise an interrupt firing during a flash operation would fault. buttonISR is declared void IRAM_ATTR buttonISR() for this reason.

The button handler is split into two stages:

  1. ISR (buttonISR) — IRAM-resident, runs on every rising/falling edge of GPIO34. Reads the current pin level, applies a debounce window, then latches three volatile variables (buttonCurrentState, buttonInterruptTime, buttonStateChanged = true) for the main loop to consume.
  2. Deferred (processButtonEvents) — called from Arduino loop() at up to 100 Hz. Reads the latched state, drives the full press/release/long-press/double-click FSM (§8.2), updates the LCD with countdown text, and enforces the 3 s inactivity timeout.
Debounce mechanism (in detail). Mechanical tactile switches bounce — a single press typically produces 5–20 rapid edges over 2–10 ms before settling. The ISR defeats this by recording the millis() timestamp of every edge it processes into buttonInterruptTime. When the next edge arrives, the ISR computes currentTime − buttonInterruptTime; if that delta is less than BUTTON_DEBOUNCE_TIME (50 ms), the edge is discarded and no state change is signalled. Only the first edge of a contact-close (or contact-open) transition survives, and the subsequent bounce edges within the 50 ms window are filtered out. 50 ms is slow enough to defeat bounce but fast enough that a human user cannot press/release faster than the window.
⚠ Do not expand the ISR. String, Serial, Wire (I²C), and Preferences calls are all unsafe from IRAM — they may touch non-IRAM code, call malloc, or block on a bus — any of which from interrupt context can deadlock or fault.

4.8Task Watchdog strategy
  • esp_task_wdt_init(60, true) — 60 s timeout, panic on expiry.
  • Three tasks registered: main loop, mqttListenerTask, dataPipelineTask.
  • Main loop feeds ≥ every 5 s.
  • Core 0 feeds every iteration (10 ms).
  • Core 1 feeds every iteration (100 ms).
  • DataLogger writes throttled with 5 s back-off after failure — cannot chain into WDT timeout.

6.1Sensor acquisition
6.1.1 Modbus RTU read sequence

For each sensor i ∈ {1, 2}:

  1. Flush. while (Serial2.available()) Serial2.read() drains residual bytes.
  2. Transact. readHoldingRegisters(0x0000, 2) returns humidity in word 0, temperature in word 1.
  3. Scale. Both values divided by 10.0f.
  4. Offset. tempN_offset / humN_offset are added.
  5. Store. Written into sensorDataArray[i-1].
  6. Inter-frame gap. 500 ms delay() between sensor 1 and sensor 2.
6.1.2 Scaling & calibration offset
published_temp = (raw_register_word / 10.0f) + tempN_offset
published_hum  = (raw_register_word / 10.0f) + humN_offset
6.1.3 Failure handling — sentinel vs LKG

  • Four file-scope float variables in Network.cpp.
  • Initialised to 0.0f at boot.
  • Single-writer: only readSensorsAndUpdateLKG() on Core 1 writes them.
  • Lock-free on reads — each is a naturally-aligned 32-bit load.

6.4Calibration sample branch

When calibrationState.isCalibrating == true, every readSensors() additionally calls processCalibrationSample():

  1. Re-read reference probe (getCalibrationSensorData()).
  2. If first valid sample, latch reference (T, H) into calibrationState.refTemp/refHum.
  3. Compute raw sensor value by subtracting the currently-stored offset from sensorDataArray[i].
  4. Accumulate temp_sum += (ref - raw), hum_sum += (ref - raw), increment sampleCount.
  5. When sampleCount == CALIBRATION_SAMPLES = 10, average, saveCalibrationValues(), LCD "Calibrated!", stopCalibrationMode().

A sensor disconnect mid-run (zero readings on the target sensor) aborts cleanly without committing partial offsets.

8.4Sampling algorithm
1. data = getCalibrationSensorData()
   if not data.valid: return  # skip this sample
2. if first valid sample:
     ref = (data.temp, data.hum)   # latch reference
3. raw_T = sensorDataArray[i].temp - calibrationValues.tempN_offset
   raw_H = sensorDataArray[i].hum  - calibrationValues.humN_offset
4. if raw_T == 0.0 && raw_H == 0.0:   # sensor disconnected
     immediateDisplayUpdate("Cal Aborted!", "Sensor Disconn.")
     stopCalibrationMode(); return
5. calibrationState.temp_sum += (data.temp - raw_T)
   calibrationState.hum_sum  += (data.hum  - raw_H)
   calibrationState.sampleCount++
6. if sampleCount == CALIBRATION_SAMPLES (10):
     offset_T = temp_sum / 10
     offset_H = hum_sum  / 10
     calibrationValues.tempN_offset = offset_T
     calibrationValues.humN_offset  = offset_H
     saveCalibrationValues()
     immediateDisplayUpdate("Sn Calibrated!", "T:… H:…")
     stopCalibrationMode()
Two subtleties: (1) The current offset is subtracted from sensorDataArray[i] to recover the raw reading — this lets subsequent calibrations converge instead of stacking.   • __attribute__((packed)) on both structs guarantees no implicit padding.
  • ESP32 is little-endian. External tooling parsing the log must respect this.

9.4CSV export

9.5Wear-levelling
  • SPIFFS provides block-level wear levelling internally.
  • Circular ring pattern distributes erases evenly across all 10 080 slots.
  • At 1 write/min, each slot sees an erase ≈ every 7 days — ≈ 520 erases/slot over 10 years, within NOR flash endurance (typically 100 000).
  • NVS also wear-levels internally; the two high-frequency writers (OTA state, async config) are small and infrequent.

10.1Watchdogs
10.1.1 Task WDT
  • Initialised in setup(): esp_task_wdt_init(60, true) — 60 s timeout, panic on expiry.
  • Three tasks registered: main loop, mqttListenerTask, dataPipelineTask.
  • Any task that fails to call esp_task_wdt_reset() within 60 s causes the chip to reset. Last-line recovery for silent hangs.
10.1.2 MQTT keep-alive

client->setKeepAlive(60). Firmware ticks client->loop() every 10 ms when connected, and forces a tick every 15 s. A self-inflicted disconnect is visible in the serial log and triggers the normal reconnect flow.

10.1.3 Gateway watchdog

SystemHealth::checkSystemHealth() tests gateway reachability. Sustained loss triggers performPreventiveRestart("Gateway unreachable"). Defeats the observed ESP32 condition where DHCP renews correctly but the LWIP default route is missing.

10.2Heap monitoring
ThresholdValueBehaviour
HEAP_WARNING_THRESHOLD90 KB freeLogged; no action
HEAP_CRITICAL_THRESHOLD80 KB freeperformPreventiveRestart("Critical heap exhaustion")

Sampled every HEAP_CHECK_INTERVAL = 60 s. A device below 80 KB is unlikely to recover without a reboot; no heuristic "free the cache" step is attempted.

10.3Preventive periodic reboot
  • RESTART_INTERVAL_HOURS = 6. On every checkSystemHealth() pass, if uptime exceeds this, performPreventiveRestart("Scheduled 6-hour restart") is called.
  • Pre-emptive trigger: if uptime is within 30 min of the scheduled reboot and free heap is low (below HEAP_WARNING_THRESHOLD), the reboot is brought forward — code-verified at SystemHealth.cpp:213-217.

Empirically, 6 h is short enough that long-run LWIP fragmentation doesn't dominate; 24 h was observed to risk heap starvation under heavy Modbus TCP poll load.

10.4Ethernet link-loss handling
  • Link state observed via EthernetEvent() (renamed from WiFiEvent).
  • Link-down alone does not trigger a reboot; sets eth_connected = false.
  • Sustained loss feeds into the gateway watchdog and eventually triggers a restart.
ENABLE_NETWORK_STACK_RESTART = false — the firmware does not attempt to reset the LWIP stack in-place. Experience has shown a full reboot is more reliable than a partial stack reset on ESP32-Arduino 2.x.
10.5OTA-safe pausing

Before an OTA starts flashing:

  1. disconnectMQTTForOTA() calls client->disconnect() cleanly.
  2. stopDataPipeline() calls vTaskSuspend(dataPipelineTaskHandle) — Core 1's pipeline is frozen at an arbitrary instruction.
  3. AsyncOTA then owns flash exclusively.

If OTA fails without rebooting, resumeDataPipeline() calls vTaskResume(). If OTA succeeds, the device reboots into the new partition.

10.6Modbus TCP readiness gate

Until both LKG temperatures have been populated at least once, initModbusTCP() is not called. DCIM clients receive TCP RST (ECONNREFUSED) — the correct signal for "poll target not yet ready", much better than silently serving 0.0 °C.

10.7PubSubClient dead-socket workaround (ethClient.stop() pre-reconnect)

The PubSubClient library, combined with the Arduino WiFiClient glue used by the ESP32 Ethernet stack, has a field failure mode:

  1. The broker's TCP session is terminated (broker crash, carrier NAT timeout) in a way that the client does not immediately detect.
  2. The next client->connect() internally reuses the half-dead socket.
  3. The TCP handshake silently fails with false return — and every subsequent reconnect repeats indefinitely.

Workaround applied in mqttListenerTask(): call ethClient.stop() immediately before every client->connect() attempt, with a 100 ms delay after:

ethClient.stop();
vTaskDelay(pdMS_TO_TICKS(100));
client->setServer(mqtt_broker_str, mqtt_port);
client->setCallback(mqttCallback);
client->setKeepAlive(60);
bool connected = client->connect(clientId, …);
⚠ Do not remove this block. It is explicitly commented in source. It defeats the silent-reuse path.
10.8LCD nibble-sync recovery
  • The PCF8574 backpack operates in 4-bit nibble mode. Long-running operation can leave the nibble counter out of phase, producing garbled characters.
  • Every LCD_NIBBLE_SYNC_INTERVAL = 36 000 000 ms (≈10 h), performNibbleSyncRecovery() re-issues the LCD 4-bit initialisation sequence, forcing the controller back to known state.
  • Non-destructive to screen content; any loss of text is masked by the next 1 Hz updateLCD().

11.1First-power-on checklist
  1. Physical mount. Install in the rack location defined by the site drawing.
  2. Plug probes. Sensor 1 → front of rack, Sensor 2 → rear of rack. Verify cable IDs (1 and 2) match the Modbus RTU slave IDs.
  3. Plug PoE. Device boots within 3–5 s; LCD backlight comes on within 2 s.
  4. Wait for link. LCD cycles through boot messages and lands on a readings display once link is up and both sensors have responded at least once.
  5. Verify IP. Expected static IP from the site plan appears on the LCD. If rack uses DHCP, the leased IP appears.
  6. Verify FM presence. The FM dashboard shows the device within 30 s (first telemetry) or ~60 s (full registration).
  7. Note the device name. If it still shows the compile-time default device name, update it via the web UI before leaving site.
11.2Web UI walkthrough

Browse to http://<device-ip>. Default basic-auth credentials are set at provisioning.

11.2.1 Status page
  • Current T/H for both sensors; uptime; firmware version; IP config; MQTT status; Modbus TCP state; DataLogger status; time source and TimeSyncState.
11.2.2 Network config
  • Device name, DHCP toggle, static IP / subnet / gateway / DNS. Saving triggers a 3 s countdown and reboot.
11.2.3 MQTT hierarchy
  • Seven fields (country, city, DC, building, floor, datahall, device name). Saving rebuilds mqttBaseTopic and all derived topics, invalidates the hierarchy cache, and re-subscribes on next reconnect.

  • Auth: change web-UI username / password. NVS-backed, independent of MQTT HMAC keys.

11.4Performing a calibration in the field
Precondition: reference probe connected to UART1.
  1. Verify the reference probe is visible — isCalibrationSensorConnected() returns true.
  2. Single-press button. LCD: Cal Mode: S1 / Double=Start.
  3. If needed, single-press again to toggle to S2.
  4. Double-click to begin sampling. LCD: Calibrating Sn / Sample 1/10.
  5. Wait ≈100 s (10 samples × 10 s). LCD advances to Sample 10/10, then Sn Calibrated! / T:ΔT H:ΔH briefly.
  6. Device returns to idle display; offsets persisted.

To abort during select: long-press or wait 3 s. To abort during sampling: unplug the sensor being calibrated (clean abort, button ignored mid-run).

11.5OTA update procedure
11.5.1 Fleet-initiated (normal case)
  1. Platform engineer uploads new firmware to FM's artifact store.
  2. FM publishes signed UPDATE_FIRMWARE:<hmac>:<json> on the appropriate hierarchy topic.
  3. Each in-scope device computes its OTA stagger (MAC-derived group × per-group offset) and queues the download.
  4. When its stagger elapses, the device disconnects MQTT, suspends the pipeline, downloads, verifies, reboots.
  5. On next MQTT connect after reboot, publishes …/OTA_RESULT with success or failure reason.
11.5.2 Manual (on-site)
  1. Web UI → OTA page.
  2. Paste firmware URL and SHA-256 checksum.
  3. Click "Start OTA". Progress shown live. Device reboots on success.
11.5.3 What to do if OTA fails
  • Check [ASYNC_OTA] serial trace for the exact failure state.
  • SHA-256 mismatch → artifact wrong or corrupted in transit.
  • Slow-write thresholds tripped → flash may be wearing out; escalate.
  • 3 attempts all fail → device stays on old firmware, broadcasts OTA_BCAST_FAIL, continues normal operation. No bricking has occurred.

11.8Factory-reset procedure
⚠ Factory reset wipes: network config, MQTT hierarchy, auth, calibration offsets, chart prefs, OTA state.
Factory reset does not wipe: SPIFFS log files, timemanager NVS namespace.
Via front-panel button (recommended when web UI unreachable)
  1. Ensure no reference probe is connected to UART1.
  2. Press and hold the button for ≥ 1 s.
  3. LCD shows Factory Reset / Release in Ns as countdown progresses.
  4. At Release to confirm, release the button.
  5. Device clears NVS and reboots after a 3 s LCD countdown.
Via web UI

Admin → Factory Reset. Same effect.

After reset
  • Device boots with Config.h compile-time defaults (the static IP, subnet, gateway, DNS, device name, and MQTT hierarchy compiled into the firmware image). These values are deployment-specific and set at build time.
  • Re-provision via DHCP-on / FM push / web UI as your process prescribes.

FBuild & flash instructions
  1. Install Arduino IDE (2.x) or Arduino CLI.
  2. Add ESP32 board package URL: https://raw.githubusercontent.com/espressif/arduino-esp32/release/v2.x/package/package_esp32_dev_index.json.

  5. Board: ESP32 Dev Module. Partition scheme: Default (2 × OTA + SPIFFS).
  6. Open Data_Gathering.ino. Build → Upload.
  7. Bump FIRMWARE_VERSION in Config.h for every released build.

KAsyncConfigUpdate FSM reference

Enum values (Network.h): CONFIG_IDLE, CONFIG_SECURITY_CHECK, CONFIG_DEVICE_NAME, CONFIG_IP, CONFIG_SUBNET, CONFIG_GATEWAY, CONFIG_DNS, CONFIG_COMPLETE, CONFIG_ERROR.
